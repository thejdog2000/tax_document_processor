# Task: Per-Office Output Hierarchy Customization

## Goal

Allow each tax office to configure the client/year folder and file structure created by the app.

## Context

Different offices maintain different per-client/per-year directory standards. Front-office staff need the app to create the same structure consistently for every client without manually creating or renaming folders.

The current default output structure creates:

```text
Client_2025/
├── SD/
├── Review/
├── Return/
├── Signature Pages/
├── logs/
│   └── YYYYMMDD_HHMMSS_document_log.txt
└── document_log_latest.txt
```

## Scope

- Provide a settings/config UI for output hierarchy templates.
- Make `File & Folder Organization` the first settings tab.
- Allow folder creation, renaming, relocation, and deletion.
- Allow files generated by the app to be assigned to target folders.
- Support drag-and-drop reordering of folders/files in the hierarchy editor.
- Support office-level presets that can be reused across clients and years.
- Keep sensible defaults for offices that do not customize.
- Keep the per-client output destination on the main workflow screen, not buried in settings.
- Configure naming patterns for renamed PDFs.
- Configure naming patterns for generated Excel review workbooks.
- Configure which default folders are created for each client/year.
- Do not include zip output in the hierarchy editor.

## Config UI Requirements

- Show the default hierarchy as an editable tree/list.
- Provide an obvious drag handle on each row.
- Each row should show type: folder, generated file, or app-maintained log.
- Folder rows should support enable/disable, rename, and destination nesting.
- Generated file rows should support destination folder and filename pattern.
- Provide `Add Folder`, `Add Generated File Rule`, and `Restore Defaults` actions.
- Treat `logs/` and `document_log_latest.txt` as packet-facing logs, not internal diagnostics.
- Keep internal app diagnostics out of this screen unless a support/admin tab is added later.

## Default Naming Patterns

- Client folder: `{last}_{first}_{tax_year}` with blank names falling back to `Client_{tax_year}`.
- Renamed source PDFs: `{form_type}_{payer}_{tax_year}.pdf`.
- 1040 workbook: `{client_slug}_1040.xlsx`.
- DoubleCheck workbook: `{client_slug}_DoubleCheck.xlsx`.
- Latest packet log: `document_log_latest.txt`.
- Versioned packet log: `{YYYYMMDD_HHMMSS}_document_log.txt`.

## Acceptance Criteria

- Staff/admin can define a folder structure once and reuse it.
- Generated files land in the configured locations.
- The app previews the output hierarchy before processing.
- The default structure remains available.
- Configuration is stored outside the install directory and survives app updates.
- The editor never suggests or displays zip package output.
- Changing the hierarchy does not change where the user selects the per-client output destination.

## Open Questions

- Should hierarchy customization be global per office, per workstation, or per client?
- Should non-admin staff be allowed to edit hierarchy templates?
- Should output hierarchy support conditional folders based on return type or filing status?
- Should app diagnostic logs ever be configurable, or always stay in app data?
- Should office presets be import/export files so a firm can share the same setup across workstations?
